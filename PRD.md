# PRD：AI 驱动的内容采集插件管理平台

> 版本：v0.1（草案，关键决策已确认，待评审）　|　生成日期：2026-07-21
> 关联文档：`report.md`（竞品与行业分析）、`outline.yaml`、`fields.yaml`
> 技术栈（待确认）：PHP（Laravel 11/12）后端 + Vue3 前端 + MySQL

## 1. 背景（Background）

产品团队计划搭建一个**内容采集 AI 插件管理平台**：把"按网站爬取内容与链接"封装为可上架、可安装、可启停的**插件**，由用户选模型、设参数、点运行并实时看日志；管理员在后台管理用户、配置 AI 技能、对插件上下架；开发者可基于 PHP 框架上传自己的采集插件。

**为什么要做（调研支撑，详见 report.md）**
- 纯"爬虫工具"已高度内卷（Bright Data / Apify 占据基础设施层），差异化空间在 **"插件化 + AI 处理 + 低门槛建模"** 的组合。
- 最值得对标的产品形态是 **Apify（Actor 市场）+ n8n（节点即插件、可自托管）**：前者证明"采集能力 = 可交易插件"，后者证明开发者对"私有化 + 插件式"的偏好。
- **接口标准建议对齐 MCP**（Linux Foundation 旗下事实标准，3000+ Server）：零成本接入主流 Agent 生态，且天然带"Server 互不可见、Host 控边界"的隔离语义，正好服务插件沙箱。
- **安全与合规是生命线**：用户可"上传插件 + 带 Cookie 采集"，必须前置容器/子进程级沙箱、上架安全扫描、GDPR/ToS/robots 合规底线。

## 2. 目标（Goals）

| 类型 | 目标 | 衡量口径 |
|------|------|----------|
| 业务目标 | 建立"采集插件市场"雏形，激活开发者上传与用户安装 | 上线 60 天内 ≥ 20 个上架插件、≥ 100 个注册用户、≥ 500 次插件运行 |
| 产品目标 | 用户 3 步内完成"选插件→设参→运行→看日志" | 新用户首次成功运行任务的中位时长 ≤ 5 分钟 |
| 技术目标 | 插件可扩展、用户可上传、运行隔离可控 | 插件上传到可用平均时长 ≤ 2 分钟；单插件故障不拖垮主进程（100% 隔离） |
| 安全目标 | 用户上传代码与带 Cookie 采集不引发越权/泄漏 | 上架安全扫描覆盖率 100%；Cookie 密文存储率 100% |

## 3. 用户故事（User Stories）

**管理员（Admin）**
- 作为管理员，我希望在后台管理用户账号与角色，以便控制谁能安装/上传插件。
- 作为管理员，我希望配置平台的 AI 技能（可用模型/默认参数），以便统一治理模型能力。
- 作为管理员，我希望对插件进行上架/下架/审核，以便保证市场质量与安全。

**普通用户（End User）**
- 作为用户，我希望从市场安装插件并启停，以便按需使用采集能力。
- 作为用户，我希望设置插件参数（网址/Cookie/失效日期/模型/深度）后一键运行，并在右侧实时看到日志，以便掌握进度与结果。
- 作为用户，我希望修改自己的密码，以便保障账号安全。

**插件开发者（Developer）**
- 作为开发者，我希望基于平台提供的 PHP 模板开发插件并上传，以便分发我的采集能力。
- 作为开发者，我希望插件展示标题/开发者/版本/更新日期/说明，以便用户了解与信任。
- 作为开发者，我希望平台提供开发模板与接口规范，以便降低开发门槛。

## 4. 功能清单（Feature List）

### 4.1 管理后台 — 用户管理
- 用户列表（增删改查、搜索、分页）
- 角色与权限（管理员 / 普通用户 / 开发者；基于 RBAC）
- 账号状态（启用/禁用）、密码重置

### 4.2 管理后台 — AI 技能配置
- 模型供应商管理（GLM / DeepSeek / GPT 等，OpenAI 兼容协议）
- 平台级默认模型、默认参数、配额策略
- "AI 技能"概念：将某模型 + 某套提示/处理规则封装为可分配给用户的"技能"

### 4.3 管理后台 — 插件上下架
- 插件审核队列（待审 / 已上架 / 已下架 / 违规）
- 上架/下架/驳回操作，驳回须填原因
- 插件安全扫描结果查看（见 §4.8）

### 4.4 用户后台 — 插件安装与启停
- 从插件市场浏览/搜索/安装
- 已安装插件列表：启用 / 停用 / 卸载 / 查看详情
- 修改密码（强制校验旧密码、强度策略）

### 4.5 插件定义（核心概念）
- **一个插件 = 一个目标网站（或一类同构网站）的采集器**
- 职责：爬取网站**内容**与**链接**，产出结构化结果
- 插件通过平台接口规范（建议 MCP 风格）声明：采集动作（Tools）、结果数据集（Resources）

### 4.6 插件参数
| 参数 | 说明 | 约束 |
|------|------|------|
| 网址 | 目标站点 URL | 必填，格式校验；须声明目标域 |
| 登录 Cookie | 用于需登录态的站点 | AES-256 加密存储；必填失效日期 |
| 失效日期 | Cookie/任务有效期 | 运行前校验，过期阻断 |
| 使用模型 | 处理/理解内容的 AI 模型 | 从已配置模型中选择 |
| 爬取深度 | 链接层级深度 | 正整数，默认 1，设上限防失控 |

### 4.7 模型管理
- 模型的增 / 改 / 删（名称、供应商、base_url、api_key、model 名、配额）
- 支持主流 **GLM / DeepSeek / GPT**（均 OpenAI 兼容，仅 base_url + model 不同）
- **测试按钮**：以最小 payload 发一次补全请求，回显延迟 / 返回内容 / token 用量 / 错误码

### 4.8 插件协议与开发（扩展性核心）
- 定义 **PluginManifest**：插件元信息 + 钩子点（onBeforeCrawl / onAfterParse / onError）+ 权限声明（网络/文件系统/命令执行）
- 插件运行时：**独立子进程/容器执行 + 资源配额 + 权限白名单 + 危险函数禁用**（PHP 无 autoload 级隔离，必须分进程）
- **接口对齐 MCP**（建议）：插件抽象为 MCP Server，暴露 Tools/Resources/Prompts；Host = 平台管理/模型层；可零成本接入 Claude/Cursor 等生态
- **开发模板**：平台提供 PHP 插件脚手架（目录结构、manifest 示例、钩子示例、解析规则示例）
- 上架展示字段：标题、开发者、版本号、更新日期、说明
- 上架流程：**自动安全扫描（PCP 式静态检查）+ 人工审核 + 2FA**（参考 WordPress 插件生态）

### 4.9 使用流程与实时日志
- 用户流程：**选择/安装插件 → 设参数 → 选模型 → 点"运行" → 右侧面板实时看日志**
- 日志通道：推荐 **SSE（EventSource）+ Pinia**，单向流、自动重连、零额外依赖；仅当需前端反向控制（暂停/中止）再升级 WebSocket
- 每次日志同时**落库 + 推流**，刷新后可回溯

## 5. 流程说明（Flows）

**主流程 — 用户运行插件**
```
[用户] 安装插件 → 进入运行页 → 填参数(网址/Cookie/失效日期/模型/深度)
  → 选模型 → 点[运行]
[系统] 校验参数 → 校验 Cookie 未过期 → 启动独立子进程/容器
  → 调用爬虫(静态:Guzzle+DomCrawler / JS渲染:Panther)
  → 逐行产出日志(SSE 推右侧面板) → 内容落库 → 任务结束(event:done)
```

**管理流程 — 插件上架**
```
[开发者] 基于模板开发 → 打包上传 → 填元信息(标题/开发者/版本/说明)
[系统] 触发自动安全扫描 → 生成扫描报告
[管理员] 查看扫描报告 → 人工审核(通过/驳回) → 上架/下架
```

**模型测试流程**
```
[管理员/用户] 模型管理 → 选模型 → 点[测试]
[系统] 用 ChatAdapter 发最小请求(stream:false) → 回显延迟/内容/token/错误码
```

## 6. 交互说明（Interactions）

- **布局**：用户后台左侧为功能导航（市场 / 我的插件 / 模型 / 账号），右侧主区为操作/详情；**运行页采用"左参数 + 右日志"双栏布局**，日志区支持级别筛选（info/warn/error）与进度百分比。
- **表单校验**：网址格式、Cookie 必填失效日期、爬取深度正整数且 ≤ 上限、模型必选；校验失败就地提示。
- **实时性**：日志区滚动锁定最新、支持暂停滚动；任务异常标红并给出错误码与建议。
- **权限边界**：普通用户仅见自己安装的插件与自己的 Cookie；管理员可见全量但不可冒用用户 Cookie。
- **开发模板入口**：用户/开发者后台提供"开发插件"入口，一键下载 PHP 脚手架与 manifest 示例。

## 8. 验收标准（Acceptance Criteria，EARS 原则）

### Ubiquitous（系统始终满足的约束）
- 系统须对用户密码与用户 Cookie 以 **AES-256 加密**存储，密钥来自 `APP_KEY`，不得明文落库。
- 系统须为每条 Cookie 绑定 `user_id/tenant_id`，查询强制带租户条件，杜绝跨租户串号。
- 系统须在执行任意插件前校验其 **Cookie 失效日期**与**权限声明**，越权能力一律拒绝。
- 系统须对所有上架插件执行**自动安全扫描**，未通过扫描不得进入人工审核队列。

### Event-driven（由事件触发的行为）
- 当用户点击"运行"按钮时，系统须先完成参数校验与 Cookie 过期校验，校验通过后启动独立子进程/容器并开始采集。
- 当开发者上传插件时，系统须自动触发安全扫描并生成扫描报告，随后将插件置为"待审核"状态。
- 当管理员在模型管理点击"测试"时，系统须用统一 `ChatAdapter` 发送一次最小补全请求并回显延迟/内容/token/错误码。
- 当插件产出一行日志时，系统须同时写入日志表并向该任务通道推送 SSE 消息。

### Unwanted（异常 / 故障处理）
- 若目标网站返回 **429 / 被封禁**，则系统须按限速策略退避重试，并在日志区标红提示，超限则终止任务。
- 若 Cookie 已**过期或 401 登录失效**，则系统须阻断任务、标记插件/参数为"失效"并提示用户刷新。
- 若插件进程**超时或崩溃**，则系统须隔离该进程、回收资源，并将任务标记为"失败"，不影响其他任务。
- 若安全扫描发现**危险函数调用或越权声明**，则系统须直接驳回上架并给出风险项清单。

### State-driven（特定状态下的行为）
- 当任务处于"运行中"状态时，系统须持续向右侧面板实时推送进度与日志，直至收到 `done` 事件。
- 当插件处于"已下架"状态时，系统须禁止新用户安装，已安装用户须收到停用提示。
- 当租户处于"禁用"状态时，系统须暂停其全部进行中的采集任务。

### Optional（可选功能）
- 若平台启用 **WebSocket** 而非 SSE，则系统须支持前端反向控制（暂停/中止）任务。
- 若插件声明需要**命令执行权限**，则系统须在该能力经管理员显式授权后才允许执行，且仍受沙箱约束。
- 若平台对接**外部 MCP 客户端**（Claude/Cursor 等），则系统须将插件作为 MCP Server 暴露，供外部调用。

## 9. 边界场景 / 异常 / 权限 / 数据口径（补充）

- **边界**：爬取深度为 0 或超大值 → 默认 1 / 截断到上限；网址为列表而非单站 → 一期仅支持单站，多站拆多插件。
- **异常**：目标站改版导致解析失败 → 插件返回结构化错误，任务标红不崩；网络中断 → 断点续爬（一期可仅重试）。
- **权限**：管理员不可读取/冒用用户明文 Cookie；开发者仅能管理自己上传的插件；RBAC 细粒度到"安装/上传/审核/模型配置"。
- **数据口径**：版本号遵循语义化版本（递增）；"更新日期"取插件包内 manifest 声明值；运行次数按"一次点击运行 = 一个任务"计数。
- **合规底线**：插件市场默认禁止采集登录态/个人敏感数据，上架须声明目标域与数据类别；强制尊重 robots.txt、限速、提供退出机制；用户后台显著提示 GDPR/CFAA/ToS 风险（详见 report.md §六）。

## 10. 关键决策（已确认，2026-07-21）

> 以下 6 项原"待确认问题"已由产品负责人确认，作为一期范围基线。

| # | 决策项 | 确认结论 | 对 PRD 的影响 |
|---|--------|----------|----------------|
| 1 | 目标用户 | **开发者为主**（社区驱动上架，最契合"插件市场"定位） | §4.8 强化 PHP 开发模板/SDK；可视化（小白）列入二期 |
| 2 | 部署形态 | **私有化自托管为主**（PHP+MySQL 易部署、数据不出域） | §4 / §9 默认单实例私有化；多租户 SaaS 列入二期 |
| 3 | 沙箱隔离级别 | **独立子进程**（proc_open + disable_functions + 权限白名单） | §4.8 / §8 Ubiquitous 明确"子进程"为默认隔离；容器作高风险插件兜底 |
| 4 | 一期范围 | **单机单租户 + 手动运行** | 暂不做调度/多租户/市场分成；先验证"安装→设参→运行→看日志"闭环 |
| 5 | MCP 对齐深度 | **仅内部 MCP 风格接口**（不接外部客户端） | §4.8 用 MCP 范式设计插件协议（Tools/Resources/Prompts），一期不暴露给 Claude/Cursor |
| 6 | 计费模式 | **市场免抽成 + 增值/服务变现**（学 WordPress 冷启动） | §7/商业默认免抽成；按量/事件（学 Apify）列入成熟期 |

**待办（一期范围内仍需细化，非阻塞）**：开发者模板的钩子点清单、子进程沙箱的资源配额默认值、私有化部署的镜像/环境要求 —— 建议在阶段三评审中由研发补全。
